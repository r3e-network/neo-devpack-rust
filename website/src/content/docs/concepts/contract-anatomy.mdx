This page walks through a complete NEP-17 token contract, explaining every structural element from imports to generated boilerplate.

## The Full Contract

```rust
use neo_devpack::prelude::*;

neo_manifest_overlay!(
    r#"{
    "name": "SampleNEP17",
    "supportedstandards": ["NEP-17"]
}"#
);

const TOTAL_SUPPLY: i64 = 1_000_000;

#[neo_contract]
pub struct SampleNep17Contract;

#[neo_contract]
impl SampleNep17Contract {
    #[neo_method(safe)]
    pub fn total_supply() -> i64 {
        TOTAL_SUPPLY
    }

    #[neo_method(safe)]
    pub fn balance_of(account: i64) -> i64 {
        balance_of_internal(account)
    }

    #[neo_method]
    pub fn transfer(from: i64, to: i64, amount: i64) -> bool {
        transfer_internal(from, to, amount)
    }

    #[neo_method(name = "onNEP17Payment")]
    pub fn on_nep17_payment(_from: i64, _amount: i64, _data: i64) {}
}
```

## Imports

```rust
use neo_devpack::prelude::*;
```

The prelude re-exports everything a contract needs: the procedural macros (`#[neo_contract]`, `#[neo_method]`, `#[neo_event]`), the `neo_manifest_overlay!` macro, SDK types, and syscall bindings. A single glob import keeps the boilerplate minimal.

## Manifest Overlay

```rust
neo_manifest_overlay!(
    r#"{
    "name": "SampleNEP17",
    "supportedstandards": ["NEP-17"]
}"#
);
```

This macro embeds a JSON fragment into a Wasm custom section. During translation, the `wasm-neovm` translator extracts it and merges it into the generated manifest. Here it sets the contract name and declares NEP-17 compliance. See [Manifest Overlays](/concepts/manifest-overlays/) for the full merge semantics.

## Constants

```rust
const TOTAL_SUPPLY: i64 = 1_000_000;
```

Rust constants compile to inline values in the Wasm output. The translator emits them as `PUSHINT` opcodes in NeoVM. No storage slot is allocated -- the value is baked into the bytecode.

## The Contract Struct

```rust
#[neo_contract]
pub struct SampleNep17Contract;
```

`#[neo_contract]` on a unit struct marks it as the contract entry point. The macro generates the Wasm export shims that the translator expects. A contract module must have exactly one struct annotated with `#[neo_contract]`.

## The Implementation Block

```rust
#[neo_contract]
impl SampleNep17Contract {
    // methods...
}
```

`#[neo_contract]` on the `impl` block processes each method inside it. For every `#[neo_method]`-annotated function, the macro generates:

1. A public Wasm export function with the correct name and signature.
2. Parameter marshalling code that bridges Rust types to the Wasm ABI (`i32`/`i64`).
3. ABI metadata that the translator collects for the manifest.

## Method Annotations

### Safe methods

```rust
#[neo_method(safe)]
pub fn total_supply() -> i64 {
    TOTAL_SUPPLY
}
```

The `safe` flag marks a method as read-only. Safe methods can be invoked without a transaction (no GAS cost for the caller). In the generated manifest, these methods have `"safe": true`. Use `safe` for any method that does not modify contract storage or emit notifications.

### Standard methods

```rust
#[neo_method]
pub fn transfer(from: i64, to: i64, amount: i64) -> bool {
    transfer_internal(from, to, amount)
}
```

Methods without `safe` are state-mutating by default. They require a transaction and consume GAS. The `transfer` method delegates to `transfer_internal`, which handles storage updates, balance checks, and event emission.

### Name overrides

```rust
#[neo_method(name = "onNEP17Payment")]
pub fn on_nep17_payment(_from: i64, _amount: i64, _data: i64) {}
```

The `name` parameter overrides the exported method name. Rust convention uses `snake_case`, but the NEP-17 standard requires `onNEP17Payment` as the callback name. The macro generates the export with the overridden name while keeping the Rust function in idiomatic style.

<Aside type="note">
  Callback methods like `onNEP17Payment` are invoked by other contracts during token transfers. Even if the body is empty, the method must exist for the contract to receive NEP-17 tokens.
</Aside>

## Generated Boilerplate

Behind the scenes, the `#[neo_contract]` macros generate:

- **Export shims** -- One `#[no_mangle] pub extern "C" fn` per `#[neo_method]`, with the correct export name. These are the Wasm functions the translator discovers and maps to NeoVM entry points.
- **Parameter marshalling** -- Conversion between Rust types and the flat `i32`/`i64` Wasm ABI. Complex types are passed as pointers into linear memory.
- **ABI metadata** -- Custom section data that the translator reads to populate the manifest's `abi.methods` array with parameter names, types, and return types.

The developer writes only the business logic. The macros handle the mechanical bridging between Rust, Wasm, and the NeoVM ABI.
