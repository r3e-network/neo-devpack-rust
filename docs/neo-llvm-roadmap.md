# NeoVM LLVM Frontend & Backend Roadmap

This document captures the end-to-end plan for delivering a production-ready LLVM
frontend and backend targeting the Neo N3 virtual machine, together with a
Rust-based developer framework that surfaces the new toolchain. The plan is
partitioned into phases that can be executed independently and validated with
dedicated milestones.

## Phase 0 – Foundation & Reference Material
- **Spec ingestion**: curate the authoritative NeoVM opcode tables, gas costs,
  ABI rules, NEF manifest schema, syscall catalog, and runtime constraints.
- **Toolchain inventory**: decide on the baseline LLVM release, host platform
  matrix, and dependencies (CMake toolchain, Python for generators, Rust tool
  versions).
- **Repository hygiene**: ensure the skeleton backend builds (`libNeoVMCodeGen`)
  and exposes `LLVMInitializeNeoVMTarget*` for discovery. Finalize the
  translation script that syncs opcode metadata from the upstream Neo repo.
- **Deliverables**: foundational docs updated, build instructions verified on
  Linux + macOS, automated script to refresh opcode metadata.

## Phase 1 – TableGen & MC Layer Completion
- **Instruction catalogue**: expand the autogenerated TableGen fragments to
  cover operand encodings (including multi-byte immediates, prefix lengths,
  feature gating, pseudo definitions for stack helpers).
- **Register model**: describe the evaluation stack and any synthetic register
  classes required for frame management. Generate consistent register enums,
  regclass info, and pressure sets.
- **MC components**: implement `NeoVMMCTargetDesc`, asm info, disassembler stub,
  code emitter skeleton, and register/instr info glue. Add round-trip tests that
  assemble and disassemble canonical Neo scripts.
- **Deliverables**: `llvm-mc -arch=neovm` can assemble/disassemble the opcode
  suite; TableGen regenerates cleanly with no manual scaffolding.

## Phase 2 – Core CodeGen Infrastructure
- **Target machine**: finalise `NeoVMTargetMachine`, object file lowering, pass
  configuration, and exception handling policy (NeoVM is a stack VM; disable EH
  infra until semantics are defined).
- **Subtarget & features**: wire CPU/feature parsing, ABI revision flags, and
  expose feature toggles (e.g. try/catch support, manifest hints) through
  `NeoVMGenSubtargetInfo`.
- **Asm printer**: implement proper emission of instruction bytes, immediates,
  data segments, and optional manifest metadata. Add hooks for NEF packaging
  (custom streamer or post-processing step).
- **Deliverables**: `llc -march=neovm` recognises the target, prints opcode
  streams for hand-authored MIR test cases, and the MC layer can serialise them
  without crashing.

## Phase 3 – Instruction Selection & Stackification
- **Lowering strategy**: decide on GlobalISel vs SelectionDAG as the primary
  selector. Implement pattern coverage for integer arithmetic, comparisons,
  branching, loads/stores, and syscall stubs. Emit NeoVM pseudos with stack
  annotations.
- **Stackify pass**: fully realise the stack discipline pass (operand reordering,
  metadata reconciliation, spill/fill integration with `NeoVMFrameLowering`).
- **Frame & register info**: implement frame index elimination, call lowering,
  callee-save policy (if any), and stack limit checks. Flesh out `NeoVMRegisterInfo`
  methods (reserved regs, frame register semantics, elimination helpers).
- **Deliverables**: MIR regression tests covering arithmetic+control flow pass,
  stackify produces valid sequences, and the asm printer emits coherent bytecode
  for these sequences.

## Phase 4 – Runtime Integration & NEF Emission
- **Asm streamer**: extend the asm printer or implement a dedicated streamer to
  build NEF containers (bytecode, manifest, checksum). Integrate syscall table
  references and gas metadata.
- **Runtime support**: provide helper libraries for contract entry/exit,
  manifest generation, and syscall bindings for both C++ and Rust consumers.
- **Validation harness**: integrate with NeoVM emulator/reference client to run
  generated NEF artefacts, assert state transitions, gas accounting, and
  manifest correctness.
- **Deliverables**: end-to-end pipeline from MIR -> NEF, verified against the
  official Neo VM interpreter with golden tests.

## Phase 5 – Frontend Bridge (C/C++/LLVM IR)
- **Intrinsic set**: model Neo-specific intrinsics (syscalls, manifest hints,
  interop services). Ensure Clang can surface them via headers or builtins.
- **Metadata emission**: propagate stack metadata via SelectionDAG/GlobalISel,
  ensure the stackify pass handles defaults, and add verifier diagnostics for
  mismatches between IR annotations and TSFlags.
- **Clang driver support**: add `-target neovm` toolchain hooks (data layout,
  default include paths, relocation model). Provide CMake toolchain files for
  contract builds.
- **Deliverables**: simple C/LLVM IR contracts compile through Clang -> llc or
  `clang -target neovm` and run in the NeoVM emulator.

## Phase 6 – Rust Development Framework
- **Codegen path**: decide on Rust integration strategy (e.g. MIR to LLVM IR via
  `rustc` with custom target spec, or standalone Rust frontend using ink! &
  InkWell). Define `rustc` target JSON for NeoVM (data layout, pointer size,
  ABI restrictions).
- **Bindings & SDK**: create Rust crates that wrap syscall bindings, manifest
  generation, and contract scaffolding. Expose procedural macros or attribute
  helpers to mark entry points and metadata.
- **Build tooling**: integrate with Cargo (custom build scripts) to invoke the
  LLVM toolchain, generate NEF outputs, and optionally bundle manifests. Provide
  templates and project generators for common contract archetypes.
- **Testing harness**: embed emulator integration within Rust tests (e.g.
  `cargo test` spins up NeoVM emulator, deploys NEF, asserts behaviour).
- **Deliverables**: `cargo neovm build/test/deploy` workflow with documentation,
  example contracts, and CI integration.

## Phase 7 – Optimisation & Compliance
- **Optimisation passes**: identify beneficial LLVM optimisations (stack peephole
  reductions, shared constant folding, opcode-specific canonicalisations).
- **Gas & coverage tools**: integrate gas estimators, instruction counters, and
  coverage tooling for contract developers.
- **Security audits**: validate codegen against Neo’s security guidelines (e.g.
  deterministic execution, stack depth limits, manifest correctness).
- **Deliverables**: optimisation playbook, automated gas estimation tooling,
  security review reports.

## Phase 8 – Documentation, Samples, & CI
- **Developer docs**: comprehensive guides for C++ and Rust flows, opcode
  references, ABI compliance, debugging techniques.
- **Sample projects**: curated set of contracts demonstrating arithmetic,
  storage, syscalls, manifests, and Rust integration.
- **CI/CD**: automated builds/test pipelines (TableGen regen checks, MIR/MC
  tests, emulator regression suite, Rust SDK integration tests).
- **Release packaging**: binary toolchains, Docker images, VSCode extensions, or
  other developer aids.

## Governance & Iteration Cadence
- Track each phase with GitHub project boards (backlog, in progress, done).
- Establish milestone reviews after each phase to lock interfaces and update
  documentation.
- Maintain nightly builds/tests to ensure regressions are caught early.
- Keep the opcode extraction script hooked into CI to detect upstream Neo spec
  changes promptly.

***

This roadmap is intentionally comprehensive. Each phase can be tackled in
parallel by different sub-teams but should meet its deliverables before moving
on to the next major milestone.
